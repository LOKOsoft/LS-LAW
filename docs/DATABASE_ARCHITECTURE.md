# Database Architecture

Source of truth: [`prisma/schema.prisma`](../prisma/schema.prisma). This document explains the *shape* and *conventions* of the schema; regenerate the model list from the schema file if they ever drift, don't hand-edit the model list here without checking it first.

## Engine

- **SQLite** via the `better-sqlite3` driver adapter (`@prisma/adapter-better-sqlite3`), Prisma 7 with the `prisma-client` generator (not the legacy `prisma-client-js`), output to `src/generated/prisma`.
- Single file at `prisma/dev.db`, gitignored. No external database service — this is a deliberate zero-cost, fully local constraint (see `AGENTS.md` / README).
- Singleton client at `src/lib/db/prisma.ts` — always import from there, never instantiate `PrismaClient` directly in feature code.

## Conventions

- **IDs**: every model uses `id String @id @default(cuid())`. Never use auto-increment ints.
- **Human-facing numbers**: entities users reference by number carry a separate unique string field (`Client.clientNumber`, `Matter.matterNumber`, `Invoice.invoiceNumber`) generated by app code, distinct from the `id` primary key.
- **Timestamps**: `createdAt DateTime @default(now())` on every model; `updatedAt DateTime @updatedAt` only on models that are edited after creation (`Firm`, `User` has none but `Client`, `Matter`, `DocumentFile`, `Invoice` do).
- **Soft state over deletion**: status enums (`ClientStatus`, `MatterStatus`, `DocumentStatus`, `InvoiceStatus`, `UserStatus`) model lifecycle; there is no soft-delete `deletedAt` pattern — rows are either kept with a terminal status (`ARCHIVED`, `CLOSED`, `INACTIVE`) or cascade-deleted with their parent.
- **Cascades**: child records that only make sense attached to a parent (`MatterTeamMember`, `Hearing`, `CourtCase`, `InvoiceLineItem`, `Session`, `Notification`) use `onDelete: Cascade` on the parent relation. Optional cross-links to `Client`/`Matter` (e.g. `Note`, `Meeting`, `DocumentFile`, `Expense`) are also cascade so orphan rows can't outlive the entity they're filed under; `Company.clientId` is `SetNull` since a company can exist independent of a client link.
- **Money**: `Float`, not `Decimal` — acceptable for a local single-tenant demo app; would need revisiting (`Decimal` + integer minor-units) before handling real currency at scale.
- **Free-text tags**: `tags String?` is a comma-joined string (see `Client.tags`, `DocumentFile.tags`, `Clause.tags`, `KnowledgeArticle.tags`), split with `.split(",")` at the presentation layer. Not a normalized join table — fine at current scale, would need a `Tag` model + join table if tag-based querying/filtering becomes a real requirement.

## Key relationships

- `User.role: Role` (13-value enum, `CLIENT` included) is the single source of truth for access control — see [INFORMATION_ARCHITECTURE.md](./INFORMATION_ARCHITECTURE.md) for how it maps to nav/permissions.
- `User.clientId` (`portalClient` relation, named `"PortalUser"`) links a portal login to exactly one `Client` — this is what scopes the Client Portal shell to one firm client.
- `Matter` is the hub entity: `Client` → many `Matter` → many `Hearing`/`Task`/`DocumentFile`/`Note`/`Meeting`/`Invoice`/`TimeEntry`/etc. Most modules key off `matterId` and/or `clientId` (both optional on shared collaboration models like `Note`/`Meeting`/`DocumentFile`, since those can be filed against a client directly without a matter).
- `Matter.responsibleAttorneyId` (required, single owner) vs. `MatterTeamMember` (many-to-many with a `MatterTeamRole`) — the responsible attorney is always also implied to lead the matter; team members are additional staffing.
- `CourtCase` is a strict 1:1 extension of `Matter` (`matterId String @unique`) for litigation-specific fields (case number, court type, stage) — not every matter has one.
- `DocumentFile.version` (denormalized current version int) + `DocumentVersion[]` (full history) — the file row always reflects the latest version's metadata; older versions are immutable history rows.

## Role-scoped queries

Query functions across `src/features/*/queries.ts` accept an optional `{ scopeUserId }`. When present, results are filtered to matters where the user is `responsibleAttorney` or a `MatterTeamMember` (`matterScopeFilter` in `src/features/matters/queries.ts`), and this filter is reused by every module that hangs off `matterId`. Firm-wide roles (Managing Partner, Accounts, HR, Office Manager, Administrator) call queries unscoped; fee-earner roles (Senior Partner, Partner, Associate, Junior Associate, Legal Researcher, Paralegal) call them scoped. This is a query-time authorization pattern, not row-level security — there is no DB-level RLS since SQLite has none; the scoping must be applied at every call site that reads matter-linked data for a fee-earner role.

## Adding a new model

1. Add the model to `prisma/schema.prisma` following the conventions above (cuid id, timestamps, cascade rules).
2. `npm run db:migrate` to generate and apply a migration.
3. Add `queries.ts` (reads) and, if it's mutable, `actions.ts` + `schema.ts` (Zod validation shared with the form) under `src/features/<module>/`.
4. If the model needs its own list view, add `columns.tsx` next to the queries and register the module in `src/lib/constants/nav.ts` + `permission-matrix.ts` if it should appear in navigation.
